# Book Manuscript Writer

这是一个给“写书”用的 Codex Skill。

别把它想得太玄。它干的事很实在：
你把目录、规范、样章、参考资料和目标 Word 文档放好，它按顺序去看、去判断、去起草、去审校，最后把能交稿的内容写回去。

它最适合的，不是“我随便写点内容试试”这种场景，
而是下面这种情况：

- 书的目录已经定了，但正文总写得散。
- 出版社给了规范，可写着写着就跑偏。
- 参考资料很多，最后写出来却像资料拼盘。
- 你明明写完了，但就是知道这稿子还不能交。
- 你不想每写一本书，就重新磨一遍流程。

这个 Skill 的价值，不在于“帮你多写一点”，而在于“帮你少返工很多次”。

## 它到底怎么用

最推荐的方式，不是丢一堆文件让它自己猜，而是先把你的写作工作区放整齐。

### 推荐工作区结构

你可以在本地先建一个文件夹，比如：

```text
我的书项目/
├─ 00-项目说明/
│  └─ 本书说明.md
├─ 01-目录/
│  └─ 正书目录.docx
├─ 02-写作规范/
│  ├─ 写作格式规范.doc
│  └─ 正文模板.doc
├─ 03-参考资料/
│  ├─ 参考书1.pdf
│  ├─ 参考书2.pdf
│  └─ 样章参考.docx
├─ 04-章节草稿/
│  ├─ 第1章.docx
│  ├─ 第2章.docx
│  └─ 第3章.docx
└─ 05-终稿输出/
```

不一定非得叫这些名字，但建议你至少把下面 5 类材料分开：

- 目录
- 写作规范
- 正文模板或样章
- 参考资料
- 章节 Word 文档

这样做有一个很直接的好处：
Skill 不用猜，你自己也不会乱。

### 一次标准调用长什么样

你可以直接这样说：

```text
使用 $book-manuscript-writer。
请先读取“01-目录”里的正书目录、“02-写作规范”里的格式规范和正文模板、
再读取“03-参考资料”里的材料，最后撰写第1章，并写入“04-章节草稿”里的 Word 文档。
如果原文件被占用，就另存新文件，并告诉我文件名。
```

如果你只想审稿，不想重写，也可以这样说：

```text
使用 $book-manuscript-writer。
不要重写结构，只对第1章做终审校。
重点检查措辞、逻辑、行文流畅度和 AI 味，改完后写回 Word。
```

## 它会怎么工作

这部分很重要。它不是上来就写，而是会按顺序做事。

### 第一步：先看目录

它会先确认这本书到底写到哪一层。

比如：
- 只有 `1.1`，那就不能乱加层级。
- 如果 `1.1` 下需要继续分，就必须写成 `1.1.1`。

也就是说，目录不是参考建议，而是结构合同。

### 第二步：再看规范

它会先弄清楚格式要求，比如：
- 章标题怎么写
- `本章要点` 怎么放
- 子标题能不能继续往下拆
- 段落应不应该控制长度
- 字数到底怎么算

这一步特别重要，因为很多返工都不是“不会写”，而是“写完才发现格式不对”。

### 第三步：再看模板和参考资料

模板决定文风，参考资料决定内容边界。

它会吸收这些材料里的事实、术语、表达习惯和节奏，
但不会把正文写成“根据资料整理如下”的样子。

简单说：
它会参考，但不会把幕后过程写进稿子里。

### 第四步：先判断字数口径，再动笔

这个坑很常见，所以我专门把它写死了。

`1800字左右` 到底是什么意思？

有 3 种可能：
- 每章 1800 字左右
- 每个主节 1800 字左右
- 每个小节 1800 字左右

这三种写法不是一回事。

这个 Skill 在动笔前会先判断口径，不会默认乱写。

### 第五步：开始写正文

写的时候，它遵循的是“书稿思路”，不是“聊天回答思路”。

也就是说：
- 先把结构立住
- 再把论述铺开
- 再压掉空话和套话
- 尽量不写那种一眼就能看出来是 AI 自动凑出来的句子

### 第六步：不是写完就交，而是要再审一遍

这一步是这个 Skill 最重要的地方。

它会专门再审：
- 措辞顺不顺
- 逻辑有没有断
- 段落是不是太长
- 标题和正文有没有跑偏
- 哪些句子虽然没错，但就是不适合出版
- 有没有一股很明显的 AI 味

如果发现问题，不是列个意见完事，而是直接改稿。

## 它特别强调的几件事

### 1. `1.1` 下面再分层，就必须是 `1.1.1`

这条现在是硬规则。

不要写无编号小标题，不要图省事写成“普通AI为什么不行”这种裸标题。
只要在 `1.1` 下面继续分，就用 `1.1.1`。

### 2. 不要把字数口径搞错

这是高频翻车点。

`每章节1800字左右` 和 `1.1、1.2、1.3各1800字左右`，看起来只差几个字，实际写法完全不同。

这个 Skill 会先判清楚再写。

### 3. 不要交“像是写完了”的稿

很多 AI 内容的问题，不是错，而是虚。

看上去完整，读起来却发飘。

这个 Skill 最后会专门处理这些问题：
- 空话
- 大话
- 套话
- 说明书味
- 机械整齐但没人味的句子

## 适合谁用

它特别适合：

- 写教材的人
- 写手册的人
- 写指南和方法书的人
- 手里已经有出版社规范的人
- 反复写章节、反复返工，已经有点烦的人

## 安装方式

把 `book-manuscript-writer` 这个文件夹复制到你的 Codex 技能目录里。

通常是：

```text
C:\Users\你的用户名\.codex\skills\
```

放好之后，在对话里调用：

```text
$book-manuscript-writer
```

就可以用了。

## 从哪开始最合适

别一上来就拿整本书压它。

最稳的方式是：

1. 先拿一章试写。
2. 再拿同一章做一次终审校。
3. 确认目录、字数、标题层级都没误判。
4. 再逐章推进。

这样你会很快知道这套流程顺不顺手。

## 最后一句

写书最烦的，从来不只是“没灵感”。

更烦的是：
明明快写完了，结果目录不对、字数理解错了、段落太长、味道太像 AI，最后又得全部回头收拾。

这个 Skill 干的，其实就是把这些本来最容易返工的地方，提前盯住。

它不神。
但如果你真的在写书，它大概会挺有用。
